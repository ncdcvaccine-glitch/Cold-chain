# منصة جرد معدات سلسلة التبريد (101 بلدية)

منصة ويب لجرد معدات سلسلة التبريد (الثلاجات، الفريزرات، صناديق التبريد...) الخاصة ببرنامج
التطعيم على مستوى 101 بلدية في ليبيا. تعتمد على **Firebase** (Authentication + Firestore)
ويمكن استضافتها مجاناً عبر **Firebase Hosting** أو **GitHub Pages**.

## المستخدمون

- **مشرف بلدية**: **بدون كلمة مرور إطلاقاً.** يفتح رابطاً خاصاً ببلديته فقط (رابط طويل
  وعشوائي يصعب تخمينه)، فيدخل مباشرة إلى صفحة بلديته ويضيف المراكز الصحية ومعدات التبريد
  (النوع، الماركة، الرقم التسلسلي، **السعة التخزينية باللتر**، الحالة، ملاحظات). لا يمكنه
  رؤية أو تعديل بيانات أي بلدية أخرى — الحماية مبنية على الرابط نفسه وليس على كلمة مرور.
- **إدارة (admin)**: تسجّل دخول ببريد إلكتروني وكلمة مرور (هذا الحساب الوحيد الذي يحتفظ
  بكلمة مرور)، تشاهد بيانات **كل** البلديات، تصفّيها، وتصدّرها إلى ملف **Excel**.

## هيكل المشروع
```
index.html             صفحة دخول الإدارة فقط (البلديات لا تستخدم هذه الصفحة)
supervisor.html         صفحة إدخال البيانات — تُفتح فقط عبر رابط البلدية الخاص (?key=...)
admin.html              لوحة الإدارة (عرض + تصفية + تصدير Excel)
firebase-config.js      إعدادات Firebase (عدّلها ببيانات مشروعك)
data.js                 قائمة البلديات الـ101 وأنواع المعدات وحالاتها
styles.css              التنسيق العام
firestore.rules         قواعد أمان Firestore (تتحقق من تطابق رابط البلدية مع بياناتها)
create_admin.py         سكريبت بايثون لإنشاء حساب الإدارة (مرة واحدة)
generate_links.py        سكريبت بايثون لإنشاء رابط سري لكل بلدية من الـ101
manifest.json            إعدادات تطبيق الويب التقدّمي (PWA) — اسم/أيقونة/ألوان التطبيق
service-worker.js        تخزين مؤقت يجعل واجهة التطبيق تفتح ولو الاتصال ضعيف
icons/                   أيقونات التطبيق (192px / 512px + نسخ maskable)
```

## كيف تعمل حماية البيانات بدون كلمة مرور؟
كل بلدية تحصل على رابط فريد يحتوي على رمز سري طويل، مثال:
```
https://your-site.web.app/supervisor.html?key=k9F2x7QmZpL...
```
عند فتح الرابط، تتحقق المنصة من الرمز في قاعدة بيانات `links`، وتحدد بلدية المستخدم تلقائياً.
أي عملية إضافة أو حذف تُرفض من Firestore نفسه (عبر `firestore.rules`) ما لم يكن الرمز
المرفق مطابقاً لبلدية البيانات — أي أن مشرف بلدية "X" لا يستطيع تعديل بيانات بلدية "Y"
حتى لو حاول، لأن الرمز الذي يملكه غير مرتبط بها. **لا حاجة لكتابة أي اسم مستخدم أو كلمة مرور.**

> ⚠️ نقطة مهمة: من يملك الرابط الكامل يستطيع الدخول. لذلك أرسل كل رابط حصرياً لمشرف
> البلدية المعنية (واتساب مثلاً) ولا تنشرها في مكان عام.

## 1) إعداد مشروع Firebase
1. اذهب إلى https://console.firebase.google.com وأنشئ مشروعاً جديداً.
2. من **Authentication > Sign-in method**، فعّل **Email/Password** (لحساب الإدارة)
   و**Anonymous** (تسجيل دخول مجهول، لازم لعمل صفحات البلديات بدون كلمة مرور).
3. أنشئ قاعدة بيانات **Firestore Database** (وضع Production).
4. من **Project Settings > Your apps**، أضف تطبيق ويب (</>) وانسخ بيانات الإعداد
   (apiKey, authDomain...) إلى ملف `firebase-config.js` بدلاً من القيم النموذجية `YOUR_...`.
5. من **Firestore > Rules**، الصق محتوى ملف `firestore.rules` ثم انشر القواعد.

## 2) إنشاء حساب الإدارة
```bash
pip install firebase-admin --break-system-packages
# نزّل مفتاح حساب الخدمة (Service Account) من إعدادات المشروع وضعه باسم serviceAccountKey.json
python create_admin.py
```
هذا ينشئ حساب الإدارة فقط (`admin@coldchain.ly` بكلمة مرور افتراضية — عدّلها في الملف
قبل التشغيل أو لاحقاً من Authentication > Users).

## 3) إنشاء روابط البلديات الـ101
بعد نشر الموقع (الخطوة 5)، افتح `generate_links.py` وعدّل قيمة `BASE_URL` إلى رابط
موقعك الفعلي، ثم شغّل:
```bash
python generate_links.py
```
سيُنشئ رابطاً خاصاً لكل بلدية ويحفظها في ملف `municipality_links.csv` (غير مرفوع على
GitHub). افتح الملف وأرسل كل رابط لمشرف البلدية المعنية فقط.

## 4) رفع المشروع على GitHub
```bash
cd coldchain-inventory
git init
git add .
git commit -m "إطلاق منصة جرد معدات سلسلة التبريد"
git branch -M main
git remote add origin https://github.com/<اسم-المستخدم>/<اسم-المستودع>.git
git push -u origin main
```

## 5) النشر (اختر إحدى الطريقتين)

### أ) Firebase Hosting (موصى به)
```bash
npm install -g firebase-tools
firebase login
firebase init hosting   # اختر مجلد المشروع كـ public directory
firebase deploy
```

### ب) GitHub Pages
من إعدادات المستودع على GitHub: **Settings > Pages > Source: Deploy from branch (main)**.
بعد بضع دقائق سيصبح الموقع متاحاً على رابط من نوع:
`https://<اسم-المستخدم>.github.io/<اسم-المستودع>/`

> ملاحظة: نفّذ خطوة "إنشاء روابط البلديات" **بعد** النشر، حتى تضع رابط الموقع الصحيح
> في `BASE_URL` داخل `generate_links.py`.

## التطبيق على الهاتف (PWA) — قابل للتثبيت بدون متجر تطبيقات
المنصة مهيأة بالكامل كـ **Progressive Web App**. بعد فتح رابط بلديته على الهاتف، يمكن
لمشرف البلدية تثبيت الصفحة كأيقونة تطبيق حقيقية:
- **أندرويد (Chrome):** قائمة المتصفح (⋮) > **"تثبيت التطبيق"** / **"إضافة إلى الشاشة الرئيسية"**.
- **آيفون (Safari):** زر المشاركة (⬆) > **"إضافة إلى الشاشة الرئيسية"**.

بعد التثبيت يفتح التطبيق مباشرة على صفحة بلديته (لأن الرابط مع الرمز السري محفوظ في
الجهاز بعد أول زيارة)، بدون شريط عنوان المتصفح وبأيقونته الخاصة.

> ⚠️ شرط أساسي: يجب أن يكون الموقع منشوراً عبر **HTTPS** (وليس مجرد فتح الملفات محلياً).
> كل من Firebase Hosting و GitHub Pages يوفران ذلك تلقائياً ومجاناً.

## ملاحظات أمان مهمة
- لا تضع مفتاح حساب الخدمة (`serviceAccountKey.json`) ولا `municipality_links.csv` في
  GitHub — كلاهما مدرج مسبقاً في `.gitignore`.
- إذا تسرّب رابط بلدية معينة أو أردت إلغاءه، احذف الوثيقة المقابلة له من مجموعة `links`
  في Firestore Console، وأنشئ وثيقة جديدة بنفس الاسم القديم تشير لرمز جديد، أو ببساطة
  أنشئ وثيقة جديدة برمز عشوائي جديد وأرسل الرابط الجديد لتلك البلدية فقط.
- `firebase-config.js` يحتوي على مفاتيح عامة (Web API Key) آمنة للنشر؛ الحماية الفعلية
  تأتي من **Firestore Rules** المرفقة، تأكد من نشرها قبل تفعيل المنصة.

## التوسعة المستقبلية (اختياري)
- صفحة إدارة مخصصة لتوليد/إلغاء روابط البلديات من داخل لوحة admin.html نفسها (بدل تشغيل سكريبت).
- إشعارات تلقائية عند معدات بحالة "خارج الخدمة".
- تصدير تقرير لكل بلدية على حدة من لوحة الإدارة.
